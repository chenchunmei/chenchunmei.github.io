    vuePress-theme-reco betty-chen    2017 - 2021

    redis宕机后如何快速恢复

    betty-chen 7/30/2021 redis

    # Redis宕机后,如何快速实现恢复

    # 部署方式

    • 单节点部署:只有一个节点提供服务,读写均在此节点,此节点宕机后数据全部丢失,直接影响业务.
    • master-slave(主从复制)方式部署:两个节点组成master-slave模式,在master写入,slave读取,master宕机后,需要人工手动的将slave提升为master,业务影响成都取决于手动提升master的延迟.
    • master-slave+哨兵模式:master-slave与上述相同,不同的是增加一组哨兵节点,用于实时的检查master的健康状态,在master宕机后自动提升slave为新的master,最大程度降低不可用的时间,对业务影响时间短.

    从上面的多种部署方式可以看出,提高Redis的可用性能的关键是:多副本部署+自动故障恢复,而多副本正是依赖主从复制

    # 哨兵模式的工作原理

    哨兵的工作流程主要分为一下几个阶段:

    • 状态感知
    • 心跳检查
    • 选举哨兵领导人
    • 选择新的master
    • 故障恢复
    • 客户端感知新的master

    # 状态感知

    哨兵每10都会向master节点发送Info命令,info命令返回的信息中,包含了主从关系,其中包含每一个slave的地址和端口号.有了这些信息后,哨兵就会记得这些节点的拓扑信息,在后续发生故障时,选择合适的slave节点进行故障恢复.

    哨兵除了给master发送info之外,还会向每一个master节点特殊的pubsub中发送master当前的状态信息和哨兵的自身信息,其他哨兵节点通过订阅pubsub,就可以拿到每一个哨兵发来的信息.

    这么做的目的主要2个:

    • 哨兵节点可以发现其他哨兵的加入,进而方便多个哨兵节点通信,为后续的共同协商提供基础
    • 与其他哨兵节点交换master的状态信息,为后续判断master是否故障提供依据.

    # 心跳检查

    在故障发生时,需要立即启动故障恢复机制,怎么做呢?

    每一个哨兵每隔1秒中向master,slave,其他哨兵节点发送ping命令,如果对方能够在指定的时间内响应,说明节点健康存活,如果未在规定时间内(可配置)响应,那么哨兵节点认为此节点主观下线.

    为什么叫主观下线呢

    因为当前哨兵节点探测对方没响应,很有可能是两个机器之间的网络发生故障,而master节点本身没有任何问题,此时认为master故障是不正确的.

    想要确定master节点是狗真正发生故障,就需要多个哨兵节点共同确定.

    每一个哨兵节点会向其他哨兵节点询问此master的状态,来共同确定是否还在线.

    如果超时指定数量的哨兵节点认为节点主观下线,那么才会把这个节点标记为客观下线.

    # 选举哨兵领导人

    确定这个节点真正故障后,就需要进入故障恢复阶段,如果进行故障回复,也需要经历一系列流程.

    首先需要选举一个哨兵领导人,由这个专门的哨兵领导人进行故障恢复操作,不用多个哨兵都参与故障恢复.选举哨兵领导者的过程,需要多个哨兵节点共同协商来选出.

    共识算法主要为了解决在 分布式场景在分布式场景下,多个节点如何针对某一个场景达成一致的结果.

    共识算法包括很多种,例如Paxos,Raft,Gossip算法等.

    哨兵选举领导人的过程类似于Raft算法,它的算法足够简单理解.

    • 每一个哨兵都设置一个随机超时的时间,超时后向其他哨兵发送申请成为领导者的请求.
    • 其他哨兵只能对收到的第一个请求进行回复明确
    • 首先达到多数确认选票的哨兵节点,成为领导者.
    • 如果确定回复后,所有哨兵都无法达到多数选票的结果,,那么进行重新选举,直到选出领导者为止.

    选出哨兵领导人之后,故障恢复操作都由这个哨兵领导者进行操作.

    # 选择新的master

    哨兵领导人针对发生故障的master节点,需要在它的slave节点中,选择一个节点来代替其他工作.

    这个选择新的master过程也是优先级的,在多个slave场景下,优先级按照::slave-priority配置>数据完整性>runid较小者进行选择.

    也就是说优先选择slave-priority最小值的slave节点,如果所有的slave配置相同,那么选择数据最完整的slave节点,如果数据也是一样,最后选择runid较小的slave节点.

    # 提升新的master

    经过优先级选择,选出了备选的master节点后,下一步就是进行真正的主从切换.

    哨兵领导者给备选的master节点发送slaveof no one 命令,让该节点成为master

    之后,哨兵领导者会给故障节点的所有slave发送slaveof $newmaster命令,让这些slave成为新master的从节点,开始新的master同步数据.

    最后哨兵领导者吧故障节点降为slave,并写入到自己的配置文件中,待这个故障节点恢复后,则自动成为新的master节点的slave.

    到此,整个故障恢复切换完成.

    # 客户端感知新master

    最后,客户端如何拿到最新的master地址呢??

    哨兵在故障切换完成后,会向自身节点的指定pubsub中写入一条信息,客户端可以订阅这个pubsub来感知新的master的变化通知.我们的客户端也是可以通过在哨兵节点主动查询当前最新的master,来拿到最新的master地址.

    另外,哨兵还提供了"钩子"机制,我们也可以在哨兵配置文件中配置一些脚本逻辑,在故障切换的完成时,触发"钩子"逻辑,通知客户端发生切换,让客户端重新在哨兵获取最新的master地址.

    一般来说,推荐采用第一种方式进行处理,很多客户端SDK中集成好了从哨兵节点获取最新master的方法,我们直接使用即可.

    ==总结:==

    为了保证Redis的高可用,哨兵节点要准确无误的判断故障的发生,并且快速的选出新的节点,来代替其他提供服务,这个中间的流程还是比较复杂的.
