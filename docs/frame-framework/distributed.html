# 分布式微服务

# 集中式系统

集中式系统也叫做单体应用，就是把所有程序、功能、模块集中到一个项目中，部署在一台服务器上，从而对外提供服务。

# 分布式系统

分布式系统就是将程序、功能拆分成不同的子系统，部署在多台服务器上，这些子系统协提供服务，在使用上和集中式系统一样。

集中式和分布式系统相反的两个概念：“合”，“分”

# 什么是微服务架构

分布式强调系统的拆分，微服务也是强调系统的拆分，服务务架构属于分布式架构的范畴。目前为止，微服务没有一个标准的定义。

简单地说，微服务是系统架构的一种风格，它的主旨将一个原本独立的系统拆分成多个小型服务，这些小型服务都在各自独立的进程运行，服务之间通过基于HTTP的RESTful API进行通信协作。

被拆分后的每一个小型服务都围绕着系统中的某一项业务功能进行构建，并且每一个服务都是独立的项目，可以进行独立的测试、开发和部署等； 由于各个独立的服务之间使用基于HTTP的JSON作为数据通信协作基础，所以这些微服务可以使用不同语言来开发。

# 微服务架构的优缺点

  • 微服务架构是将不同功能模块拆分成多个不同的服务，这些服务之间进行独立开发和部署，每一个服务都运行在自己的进程内，每一个服务的更新都不会影响到其他服务的运行。

  • 由于每一个服务都要独立部署，所以我们可以更准确地监控每一个服务的资源消耗情况，进行性能容量的评估，通过压力测试，也容易发现各个服务间的性能瓶颈所在。

  • 由于每一个服务独立的开发，项目的开发也比较方便，减少代码的冲突、代码的重复、逻辑处理流程也更加清晰，让后续的维护与扩展更加容易。

  • 微服务可ui使用不同的编程语言进行开发

但是在系统架构领域关于服务务也有一些争议，有人倾向与在系统设计与开发中采用微服务架构实现软件系统的低耦合，被认为是系统架构未来的方向，也给微服务加否很高的评价。

同时，对微服务架构持有反对观点。

  • 微服务架构增加了系统维护、部署的难度，导致一些功能模块或代码无法复用。

  • 随着系统规模日渐增长，微服务在一定程度上也会导致系统变得越来越复杂，增加了集成测试的复杂度；

  • 随着微服务的增多，数据的一致性问题，服务之间的通信成本等都凸显了出来；

所以在系统构架时也要提醒自己；不要为了微服务而微服务。
